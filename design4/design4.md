Design 4 uses the exact same bilinear interpolation algorithm as Design 3 — every march step compares the ray's height against a bilinearly-interpolated surface height h_interp, so silhouettes are smooth, not just the shading — and the only thing that changes is how the four corner reads are scheduled in time. Where Design 3 spreads its 4 corner reads across 2 cycles on 2 ports per step, Design 4 folds them across 4 cycles on a single port per step (issue h00, h10, h01, h11 on four consecutive cycles, each captured one cycle later). This halves the marcher's BRAM cost from 32 ports → 16 copies (Design 3) down to 16 ports → 8 copies → 16 BRAM18 tiles, plus 2 tiles for the normal module, for a total of 18 BRAMs — the cheapest of the four variants. The saving is bought purely with throughput: the same total reads per pixel, spread more thinly, so the pipeline runs at quarter-rate (1 pixel every 4 cycles) instead of Design 3's half-rate. To keep the single port free of contention, marcher4 uses a 2-bit phase counter to admit a new real pixel only on phase 0 (every 4 cycles), which guarantees that a step's 4-cycle read window never overlaps the next pixel's — this was verified in simulation with zero port collisions across a sustained quarter-rate stream. Internally each march_step4 grew to a 9-cycle latency (advance, four issue/capture sub-stages B0–B3, a B4 capture, a B5 valid/payload alignment stage, then the x-lerp, y-lerp+hit-test, and output buffer); the B5 stage exists to fix a subtle skew where the valid token reached the lerp one cycle before the fourth corner had settled into its register. The normal4 module is functionally identical to normal3 (forward-difference normal, 2 ports / 1 copy, smooth-height passthrough), and ray_gen/shader are unchanged, with the marcher's BRAM ports reverting to a flat [N_STEPS] array (16 single ports) rather than Design 3's 2-D [N_STEPS][2]. All preserved invariants from Design 3 carry over exactly — frozen-ray pattern, the Bug 2 fix, comparison against h_interp, and lerp arithmetic reused verbatim so the Python golden model stays bit-identical. The one new system-level cost is the clock: at quarter-rate the core must run at 100 MHz (4 × the 25 MHz pixel clock) to sustain scanout, versus Design 3's 50 MHz, so Design 4 trades 16 fewer BRAMs for a tighter timing target on the stage-D2 lerp path. Net effect: Design 4 is the BRAM-minimal way to get Design 3's smooth silhouettes, ideal when BRAM is the binding constraint and the 100 MHz timing closes — the exact mirror-image trade-off to Design 3, which is BRAM-heavy but timing-relaxed.
